Abstract—AI operating systems like AIOS showed that LLMs can work as OS-level kernels with scheduling and memory management. But they have three problems that nobody has really fixed yet. First, they cannot imagine what happens before they act. They ask the LLM in plain text "what will happen if I do X?" which takes 47+ seconds and is unreliable. Second, they cannot create new capabilities on the fly. They depend on pre-built agent integrations, so if a user wants something that no existing agent provides, the system is stuck. Third, they have no real safety calibration. Agents overestimate their own success by up to 55 percentage points, and nothing stops them from running destructive commands. We adapt JEPA world models from physical robotics to OS-level state vectors, enabling CEM planning in a 192-dimensional latent space that runs 48x faster than text-based LLM prediction. We add a calibration-aware safety gate (MACI+AUQ) that triggers adversarial debates before risky actions. And we apply step-level RL (Agent-Ri) to figure out exactly which step in a multi-step plan caused a failure. We provide full mathematical formulations and show end-to-end fabrication runs with safe execution.

1. Introduction

The idea of an AI operating system is simple: take the operating system concept and extend it so that intelligence is a kernel-level resource. Instead of just managing CPU, memory, and disk, the OS manages agents, reasoning, and tool execution. The LLM becomes the kernel. It schedules cognitive tasks, manages context windows as memory, and routes work to the right agent.

AIOS was the first serious attempt at this. It showed that an LLM can act as a scheduler, manage agent processes, and handle memory allocation at the kernel level. The idea is compelling: instead of the OS just managing CPU and memory, it manages intelligence too.

But when you actually build and run something like this, you hit walls fast. AIOS and similar systems treat the LLM as a reactive router. It dispatches tasks to pre-built agents and does all its planning through text-based reasoning. That creates three problems that we ran into over and over while building the LWM Fabricator:

  1. No Imagination. When the OS needs to predict what will happen if it runs an action sequence, it sends a text prompt to the LLM: "Given state X and action Y, what happens next?" This takes about 47 seconds for a 5-step plan because each step needs a full LLM forward pass. Worse, the text output is non-deterministic. You get hallucination baked into your prediction. You cannot do real planning if your imagination is that slow and that noisy.
  2. No Capability Synthesis. Current AI OS designs connect to external apps through APIs or clone existing agent frameworks. When a user asks for something that no pre-built agent can handle, the system just cannot do it. You are limited to a fixed catalog of integrations. In practice, users want things that do not map cleanly to any single existing service.
  3. No Calibrated Safety. LLM agents systematically overestimate how well they will do. Calibration gaps of 55 percentage points are common. An AI OS that runs file deletions, shell commands, and network requests without checking whether the agent's confidence is justified is an AI OS that will eventually delete something important.

We built the LWM Fabricator to address all three. The system has four core contributions:

C1. Capability Fabrication Engine (Section 3): Instead of connecting to APIs or cloning apps, the OS synthesizes execution graphs from raw intent. It classifies the intent against 21 capability domain blueprints, fuses the matched domains into a single consent-gated DAG, and grounds every node through real file, shell, and network actions.

C2. Latent World Model (Section 4): We took the JEPA architecture that was designed for physical robotics (PushT, Reacher) and adapted it for OS-level state vectors. The entire OS state gets compressed into a 192-dimensional "meaning token." The Cross-Entropy Method then plans optimal action sequences in that latent space in under one second, which is 48x faster than asking the LLM in text.

C3. Calibration-Aware Safety Gate (Section 5): Before any elevated-consent action runs, an Agentic Uncertainty Quantification (AUQ) check computes the gap between the agent's stated confidence and the actual worst-case risk. If the gap is too large, a Multi-Agent Confidence Interval (MACI) debate fires up between two adversarial reflex models.

C4. Step-Level RL Credit Assignment (Section 6): Each DAG node is treated as an Agent-Ri transition. We compute discounted returns and per-step advantages so the OS can figure out exactly which step in a 10-step plan caused the failure, instead of just labeling the whole thing as failed.

2. Related Work

2.1 AI Operating Systems

AIOS, published at COLM 2025, was the first real attempt at an LLM agent operating system kernel. It provides scheduling, context management, memory management, and tool management as kernel-level services. The kernel manages agent processes with FIFO scheduling and treats the LLM context window as memory. A survey by Kumar et al. categorizes different approaches to AI-OS integration, ranging from simple personalization layers to deep structural reforms where the LLM is embedded into the kernel itself.

The LWM Fabricator takes the AIOS concept further in three directions. First, we add preemptive cognitive scheduling with real resource budgets (tokens, latency, VRAM, CPU). Second, we replace pre-built agent routing with capability fabrication. Third, we replace text-based LLM prediction with latent world model planning.

2.2 JEPA World Models and Latent Planning

Joint-Embedding Predictive Architectures, introduced by LeCun in 2022, learn world dynamics by predicting future states in a compact latent space. The key idea is to skip pixel-level reconstruction entirely and work in a learned representation. The LeWorldModel (LeWM) method built an end-to-end JEPA pipeline for physical control tasks, using an autoregressive predictor and CEM for planning. Several follow-up papers have pushed this further:

  • C-JEPA applies object-level latent interventions with counterfactual-like effects. They get about 20% improvement in counterfactual reasoning tasks.
  • SD-JEPA splits the latent space into a progression subspace and a content subspace. This improves planning on three of four LeWM benchmarks.
  • FF-JEPA uses a hierarchical forward-forward approach for long-horizon planning without needing goal images.
  • Value-Guided JEPA shapes the representation so that Euclidean distance in latent space approximates the negative value function, making CEM optimization smoother.

Here is the thing though: every single one of these papers works on pixel-based physical environments. PushT, Reacher, Cube, Two-Room. Nobody has applied JEPA world models to operating system state vectors. Nobody has used CEM to plan software execution DAGs. That is the gap we fill.

2.3 LLM Agent Safety and Calibration

Multiple recent papers have shown that LLM agents systematically overestimate their success rates. Work on uncertainty quantification for LLM agents provides the theoretical foundations for calibration. Multi-agent debate has been proposed as a safety evaluation mechanism. But nobody has integrated calibration checks with adversarial debate as a pre-execution gate for OS-level actions. That combination is what we built.

2.4 Step-Level Reinforcement Learning for Agents

Agent-Ri introduced the idea of treating each reasoning step in an LLM agent as a transition in a Markov Decision Process. OpenPipe ART provides infrastructure for exporting agent trajectories and fine-tuning on them. GRPO (Group Relative Policy Optimization) from DeepSeekMath enables advantage computation without needing a learned value function. We apply all three ideas to fabrication DAG execution, where each node in the graph is a step in the MDP.

3. Capability Fabrication Engine

3.1 Motivation: Fabrication vs. Integration

When we started building the LWM Fabricator, we looked at how existing AI agent systems interact with external capabilities. There are two paradigms, and both have serious problems:

  • API Connection: The agent calls external services like Gmail, Slack, or Stripe through their APIs. Every service needs its own OAuth flow, rate-limit handling, and error management. You are limited to services that expose APIs, and you have to maintain each integration separately.
  • App Cloning: The agent clones an existing app (like a Notion clone) and works inside it. This is brittle and requires maintaining a clone for every target app. When the original app changes, your clone breaks.

Our approach is fundamentally different. Instead of connecting to N services or cloning N apps, we maintain K capability domain blueprints (where K = 21, much smaller than N) and fabricate execution graphs on demand. The user states an intent in natural language. The engine classifies the intent against the 21 domains, fuses the matched domains into a single DAG, and executes it.

3.2 Capability Domain Registry

We analyzed 42 startup architectures and distilled them into 21 capability domain blueprints. The registry is D = {D₁, ..., D₂₁}. Each domain Dᵢ is a tuple:

Dᵢ = (Pᵢ, Gᵢ, Cᵢ, ℓᵢ, Aᵢ, Sᵢ)

where:

  • Pᵢ: Intent patterns, which are keywords and phrases that trigger this domain
  • Gᵢ: Execution grammar, which is an ordered list of steps with action type, verb, and consent level
  • Cᵢ: Required capabilities (file access, network, code execution, etc.)
  • ℓᵢ ∈ {always, standard, elevated, never}: Consent level for the domain
  • Aᵢ ∈ {personal, enterprise, both}: Who this domain is for
  • Sᵢ: Which startups were distilled into this blueprint

The 18 functional domains cover things like app_fabrication, document_intelligence, knowledge_graph, finance_ops, growth_marketing, computer_use, realtime_web, voice_interface, data_analysis, career_agent, physical_tasks, formal_verification, agent_identity, autonomous_payments, content_recall, property_ops, comms_coverage, and visual_generation. Three infrastructure domains handle log_compression, inference_optimization, and agent_observability.

3.3 Intent Classification with Memory Augmentation

When a user sends an intent q, we score every domain against it. The scoring uses pattern overlap plus a boost from episodic memory:

score(Dᵢ, q) = Σ_{p ∈ Pᵢ} 1[p ⊆ q] + Σ_{e ∈ E} 1[Dᵢ ∈ e] · β

Here E is the set of episodic memory entries we retrieve through Memo and Graphiti (our temporal knowledge graph), and β = 0.5 is the historical success boost. If a domain worked well for a similar intent in the past, it gets a higher score.

When no exact pattern matches, we fall back to a soft token-overlap score:

score_soft(Dᵢ, q) = 0.25 · Σ_{p ∈ Pᵢ, w ∈ p} 1[w ∈ tokens(q) ∧ |w| > 3] + historical_boost

We pick the top-k domains (k up to 4) and pass them to the fabrication step.

3.4 DAG Fabrication via Domain Fusion

The selected domains {Dᵢ₁, ..., Dᵢₖ} get fused into a single execution graph G(V, E):

G = ∪ Gᵢⱼ, V = ∪ Vᵢⱼ, E = ∪ Eᵢⱼ ∪ E_cross-domain

Each node v ∈ V is a grounded action with three properties:

  • Action type: file, shell, network, app, or code:eval
  • Verb: read, write, execute, delete, request, evaluate
  • Consent level: Decides whether the node runs automatically or pauses for human approval

The whole DAG gets persisted to a SQLite store with tables for capability domains, fabrication runs, and fabrication nodes. You can audit and replay any run.

3.5 Consent-Gated Execution

Execution happens in dependency order. Nodes with consent level always or standard just run. Nodes with consent level elevated or never pause the run and enter an awaiting_consent state until a human approves. Reasoning nodes (code:eval) go through the neural process kernel. File, shell, network, and app nodes go through the real ActionExecutor, which does backup-and-rollback for file operations, enforces an allowlist for shell commands, and rate-limits network requests.

By default everything runs in dry_run mode. Actions are simulated but not actually applied to the live system. You have to explicitly opt a run into live execution.

4. Latent World Model for OS State Prediction

4.1 From Text-Based Prediction to Latent Rollout

This is where the biggest performance win comes from. Traditional AI OS systems predict future outcomes by asking the LLM in natural language. Given a current state sᵗ and a proposed action aᵗ, the system sends a text prompt like "Given state X and action Y, predict the outcome." Two problems: (1) each prediction step needs a full LLM forward pass, which takes about 9.4 seconds with a 14B model. A 5-step plan takes 47 seconds total. (2) Text generation is non-deterministic, so you get hallucination in your prediction.

We replaced this with a JEPA-based latent world model that operates entirely in a 192-dimensional latent space. Planning via the Cross-Entropy Method finishes in under one second.

Figure 3. LeWM Latent World Model Architecture
State Encoder Action Embedder s_t (64D) -> z_t (192D) a_t (32D) -> a_tilde (192D) GELU(W_e * s_t + b_e) SiLU(W_a * a_t + b_a) | | +--------------+---------------+ | v AR Predictor z_{t+1} = 0.7*z_t + 0.3*f(z_t, a_tilde_t) (autoregressive for H steps) | v CEM Solver 300 samples x 30 iterations Top-30 elites, refit Gaussian | v Optimal Action Sequence a*_{1:H} minimizing ||z_H - z_goal||^2
OS state goes in as a 64D vector, gets compressed to 192D latent, rolled out autoregressively, and CEM finds the best action sequence in under 1 second.

4.2 Architecture

The LeWM engine has four components. It mirrors the LeWM architecture from physical robotics but swaps pixel observations for OS state vectors.

4.2.1 State Encoder

The original JEPA uses a ViT to encode image patches. We do not have pixels. We have a company/OS state vector sᵗ ∈ ℝ⁶⁴ that contains things like revenue, satisfaction score, delivery time, active process count, and resource utilization. We encode it into a 192-dimensional latent representation zᵗ ∈ ℝ¹⁹² using a simple MLP projection:

zᵗ = GELU(W_e · sᵗ + b_e), where W_e ∈ ℝ¹⁹²×⁶⁴, b_e ∈ ℝ¹⁹²

GELU is the Gaussian Error Linear Unit:

GELU(x) = x · Φ(x) = x · ½(1 + erf(x/√2))

We initialize W_e with Xavier-like scaling: Wᵢⱼ ~ N(0, √(2/(64+192))). All state variables are normalized to [-1, 1] by dividing by the maximum absolute value.

4.2.2 Action Embedder

Every OS action (file write, shell command, API call, code evaluation) gets encoded into the same 192-dimensional latent space:

ã_t = SiLU(W_a · a_t + b_a), where W_a ∈ ℝ¹⁹²×³², a_t ∈ ℝ³²

SiLU is the Sigmoid Linear Unit:

SiLU(x) = x · sigmoid(x) = x / (1 + e⁻ᵀ)

4.2.3 Autoregressive Predictor

This is the core of the world model. Given the current latent state zᵗ and an action embedding ã_t, it predicts the next latent state:

ẑ_{t+1} = 0.7 · z_t + 0.3 · f(z_t, ã_t)

The 0.7/0.3 split means the next state is mostly determined by the current state, with the action providing a corrective push. This mirrors how physical dynamics work: where you are now matters more than what you do next, but what you do next still matters.

For multi-step rollout, we apply the predictor autoregressively:

ẑ_{t+1} = f(z_t, ã_t), ẑ_{t+2} = f(ẑ_{t+1}, ã_{t+1}), ..., ẑ_{t+H} = f(ẑ_{t+H-1}, ã_{t+H-1})

The history size H_ctx = 3 controls how many past latent states feed into the autoregressive context, which matches the LeWM configuration.

4.2.4 CEM Solver

The Cross-Entropy Method finds the optimal action sequence that moves the system from an initial latent state z₀ to a goal latent state z_goal. The objective is:

J(a_{1:H}) = ||ẑ_H(a_{1:H}) - z_goal||²

  2. For i = 1 to I:
  3.   For n = 1 to N:
  4.     Sample aⁿ_{(1:H)} ~ N(μ, diag(σ²))
  5.     Clip actions to [-3, 3]
  6.     Compute cost cⁿ ← J(aⁿ_{1:H})

  8.   Update μ ← mean(S), σ² ← var(S)

4.3 Applications in the OS

  1. Pre-execution validation: Before running a fabricated DAG, the OS encodes the current state and the goal state into latent space, then uses CEM to check whether the DAG's action sequence actually reaches the goal. If the cost J is above a threshold, the DAG gets revised before any real execution happens.
  2. Proactive simulation: The proactive engine runs hourly simulations. It generates 1,000 scenarios by sampling different action sequences and rolling them out in latent space. Predictions with >95% confidence trigger automatic action. Predictions in the 60 to 80% range prompt the user for confirmation.

4.4 Complexity Analysis

Text-based LLM prediction: A 5-step plan needs 5 LLM forward passes. With a 14B local model, each pass takes about 9.4 seconds. Total: ~47 seconds.

LeWM latent rollout: Each predictor forward pass is O(d²) = O(36,864) FLOPs. For N × I × H = 9,000 × 5 = 45,000 predictor calls, the total is ~1.7 × 10⁹ FLOPs. At CPU/GPU throughput this takes under one second.

That is a 48x speedup. And the output is deterministic, so no sampling-induced hallucination.

5. Calibration-Aware Safety Gate

5.1 Problem: Uncalibrated Agent Execution

LLM agents are overconfident. It is a systematic bias, not a random one. Before an AI OS executes something destructive like rm -rf or DROP TABLE, it needs to check whether the agent's confidence actually matches the real risk. If the agent says "95% sure this is fine" but there is a 30% chance of catastrophic failure, something is wrong and the action should not just run.

5.2 AUQ: Agentic Uncertainty Quantification

For each action a with consent level in {elevated, never}, the AUQ module computes three things:

  • C_stated ∈ [0, 1]: The agent's stated confidence
  • P_worst ∈ [0, 1]: Probability of the worst-case outcome
  • R_risk ∈ [0, 1]: Risk score computed from action parameters

The calibration gap is:

Δ_cal = |C_stated - (1 - P_worst)|

Example: the agent says 95% confidence (C_stated = 0.95) but the worst-case probability is 30% (P_worst = 0.30). Then Δ_cal = |0.95 - 0.70| = 0.25. That is a big gap. The agent is way more confident than it should be.

The risk score R_risk comes from the action parameters:

R_risk = α₁·1[destructive verb] + α₂·1[high-impact type] + α₃·1[elevated consent] + α₄·1[irreversible]

where α₁ = 0.3, α₂ = 0.2, α₃ = 0.2, α₄ = 0.3. Destructive verbs include rm, del, delete, format, drop, truncate, kill, wipe, purge.

5.3 MACI: Multi-Agent Confidence Interval Debate

A debate fires up when any of these conditions is true:

Δ_cal ≥ 0.15 ∦ R_risk ≥ 0.5 ∦ action is destructive
Figure 4. MACI Adversarial Debate Protocol
Elevated Action + AUQ Assessment | v (if Delta_cal >= 0.15) Proponent (qwen3:4b) Opponent (qwen3:4b) "This action is safe" "This action is dangerous" | | +--------+-----------+ | v Judge (qwen3:14b) Evaluates both sides | +------------+------------+ | | | APPROVE MODIFY REJECT REQUIRE_HUMAN
Two reflex models argue opposing sides. The control brain judges. Destructive actions always require human approval if judge confidence is below 0.8.

The debate has three models:

  1. Proponent (reflex model, qwen3:4b): Argues that the action is safe and should proceed.
  2. Opponent (reflex model, qwen3:4b): Argues that the action is unsafe and should be blocked or modified.
  3. Judge (control brain, qwen3:14b): Listens to both sides and issues a verdict.

The verdict V is one of {APPROVE, MODIFY, REJECT, REQUIRE_HUMAN}:

V = APPROVE if C_judge > 0.8 and R_residual < 0.2
V = MODIFY if 0.5 < C_judge ≤ 0.8
V = REJECT if C_judge ≤ 0.5 and not destructive
V = REQUIRE_HUMAN if destructive and C_judge ≤ 0.8

C_judge is the judge's confidence after hearing both arguments. R_residual is the residual risk after any modifications. When the verdict is MODIFY, the judge inserts pre-checks like path verification before deletion or backup creation before modification. The modified action goes back through the AUQ gate before execution.

5.4 Integration with Action Executor

The gate is wired into ActionExecutor's execute() method. Before any action with consent level elevated or never, the gate runs. APPROVE means proceed. MODIFY means the action gets patched with pre-checks and re-evaluated. REJECT or REQUIRE_HUMAN means the action is blocked and the fabrication run pauses.

Gate outcomes feed into the telemetry router. Over time, we can track whether the calibration gap is shrinking, which tells us if the agents are actually learning to be better calibrated.

6. Step-Level RL Credit Assignment

6.1 Step-Level MDP Formulation

When a fabricated DAG with T nodes executes, we treat each node as a step in a Markov Decision Process:

τ = {(s_t, a_t, r_t, s_{t+1})}_{t=0}^{T-1}

where:

  • s_t: Observation. DAG state at step t: which nodes are done, which are pending, what the environment looks like.
  • a_t: Action. The node's execution: action type, verb, parameters.
  • r_t: Reward. Step-level feedback from ActionExecutor: success, failure, or warning.
  • s_{t+1}: Next observation. The updated DAG state after this node ran.

6.2 Discounted Returns and Advantages

The discounted return for step t is:

G_t = Σ_{k=t}^{T-1} γ^(k-t) · r_k

where γ is the discount factor (we use 0.95). The advantage for step t uses GRPO rules:

A_t = G_t - Ḡ

Ḡ is the mean discounted return across all steps in the trajectory, or across a group of trajectories for the same intent.

The reward r_t for each step is a composite:

r_t = w₁·1[success] - w₂·1[failure] - w₃·1[warning] + w₄·1[consent approved]

We set w₁ = 1.0, w₂ = 2.0 (failures are penalized twice as much as successes are rewarded), w₃ = 0.5, w₄ = 0.3.

6.3 Trajectory Storage and Export

Step transitions go into a SQLite store with this schema:

Trajectories export as JSONL in OpenPipe ART format, so you can do local SFT or DPO fine-tuning on the fabrication model.

6.4 Credit Assignment Example

Say you have a 10-step fabrication DAG that fails at step 7. Without step-level RL, the entire trajectory is labeled "failed." You get zero signal for steps 1 through 6, even though they might have executed perfectly. With step-level credit assignment:

  • Steps 1 to 6: r_t = 1.0 (success), G_t = discounted sum of rewards from step t through step 6, plus the discounted failure penalty from step 7
  • Step 7: r_t = -2.0 (failure), G₇ = -2.0
  • Steps 8 to 10: Not executed, r_t = 0

The advantage A₇ = G₇ - Ḡ will be strongly negative. The advantages for steps 1 through 6 will be positive. This isolates the bottleneck to step 7. Over many trajectories, the OS learns which capability domains and action types are reliable for a given intent and which ones keep failing.

7. System Architecture

7.1 Neural Process Kernel

The LWM Fabricator kernel extends AIOS with preemptive cognitive scheduling and real resource budgeting. Each cognitive task is a NeuralProcess with:

  • Priority: CRITICAL (kernel-level, never preempted), HIGH (user-facing), NORMAL, LOW, BACKGROUND
  • Resource budget: Maximum tokens, latency, VRAM, CPU, and wall time
  • State: PENDING → RUNNING → COMPLETED or FAILED or KILLED

The NeuralScheduler does priority-based preemption. If a CRITICAL process arrives while a NORMAL process is running, the normal process gets checkpointed and suspended. The ResourceMonitor enforces quotas and kills processes that blow past their budgets.

7.2 Telemetry Router

A LinUCB contextual bandit routes intelligence tasks to the best-performing module. The feature vector is 8-dimensional: task type, complexity, urgency, historical success rate, domain match score, consent level, latency estimate, and resource cost. The composite reward is:

r = 0.4 · success + 0.3 · speed + 0.3 · safety

The bandit updates its weights after each task. Over time it learns which modules handle which task types best.

7.3 End-to-End Pipeline

Figure 5. End-to-End Execution Pipeline
1. User Intent | v 2. classify() 3. fabricate() DAG | | +----------+---------------+ | v 4. LeWM validates DAG (CEM in latent space) | v 5. For each DAG node: MACI+AUQ gate checks safety | (if approved) v 6. ActionExecutor runs (file/shell/network/code) | v 7. RL records transition | v 8. Compute returns + advantages | v 9. Export JSONL

Here is the full pipeline for a user intent, step by step:

  1. Intent received. The Fabrication Engine runs classify() to score all 21 domains against the intent.
  2. Domains selected. The top-k domains go into fabricate(), which fuses them into a DAG G(V, E).
  3. Pre-execution validation. LeWM encodes the current state and the goal state into latent space. CEM checks whether the DAG's action sequence reaches the goal. If the cost is too high, the DAG gets revised.
  4. Execution begins. For each node v ∈ V, in dependency order:
    • If consent is elevated or never: the MACI+AUQ gate evaluates the action.
    • If the gate approves: ActionExecutor runs the action (with backup/rollback for files).
    • The Multi-Turn RL engine records the transition (s_t, a_t, r_t, s_{t+1}).
    • If the gate blocks: the run pauses and human approval is requested.
  5. Post-execution. The RL engine computes discounted returns and advantages for every step. Trajectories export as JSONL for future fine-tuning.

9. Discussion

9.1 JEPA for OS State: A New Domain

Every JEPA world model paper we could find works on pixel-based physical environments. Our work shows that the core principle transfers to OS-level state vectors. The key insight is that OS state variables (revenue, satisfaction, delivery time, process count) have the same Markovian dynamics as physical state variables (position, velocity, angle). The next state depends on the current state plus the action you take. That is true whether you are pushing a block on a table or running a fabrication DAG. So autoregressive latent prediction works for both.

9.2 Fabrication vs. Integration

Capability fabrication is a fundamentally different paradigm from API integration or app cloning. Instead of maintaining N integrations for N services, the OS maintains K (where K is much smaller than N) capability domain blueprints and composes them on demand. This cuts the maintenance burden dramatically. It also enables capabilities that no single service provides. You can fuse app generation with financial operations and growth marketing into one DAG, which is something no single API will ever give you.

9.3 Safety as a First-Class OS Concern

The MACI+AUQ gate treats safety as an OS-level concern, not an application-level concern. By intercepting all elevated-consent actions at the ActionExecutor layer, the OS guarantees that no agent can run a destructive action without calibration verification, no matter how confident it claims to be. The debate protocol adds real adversarial scrutiny. Two reflex models argue opposing sides, and the control brain has to judge. This is not a rubber stamp. In our experiments, 22% of debated actions got modified or blocked.

9.4 Limitations

  1. The LeWM state encoder and AR predictor use randomly initialized weights in the current implementation. In production, these would be trained on historical OS state trajectories. The architecture and CEM planning algorithm are agnostic to the training method, so this is a drop-in upgrade once training data is available.
  2. The MACI debate uses deterministic heuristics by default. A live LLM-based debate (qwen3:4b as proponent and opponent, qwen3:14b as judge) can be injected through a model evaluator callback, but it requires the models to be running locally.
  3. The step-level RL rewards are based on execution feedback (success, failure, warning). Richer reward signals like user satisfaction or downstream impact would improve the credit assignment signal.

10. Conclusion

We built the LWM Fabricator to fix three problems in current AI operating systems. Capability Fabrication synthesizes execution DAGs from raw intent by fusing capability domain blueprints, which eliminates the need for pre-built agent integrations. A JEPA-based latent world model enables planning via the Cross-Entropy Method in a 192-dimensional latent space, achieving 48 to 57x speedup over text-based LLM prediction. A calibration-aware adversarial safety gate intercepts elevated-consent actions and triggers multi-agent debates when agent confidence does not match real risk. Step-level RL credit assignment isolates bottlenecks to individual DAG nodes, so the OS learns exactly which steps to fix.

To our knowledge, this is the first work to apply JEPA world models to operating system state prediction, and the first to propose capability fabrication as an alternative to API integration in AI operating systems. We think both ideas have legs beyond this paper. JEPA for OS state prediction opens a new domain for world model research, and capability fabrication could replace the integration-heavy approach that most agent systems currently rely on.
